# 2026-07 Monthly Plan

## Scope

기간: 2026-07-01 ~ 2026-07-31

7월 목표는 6월 P0 PoC thin-slice를 기반으로 **관제 MVP를 안정화하고 실제 로봇팀 연동 전 단계까지 끌어올리는 것**이다.

핵심 방향:

- 6월 PoC 기능의 안정화와 회귀 방지
- 녹화/리플레이 운영 안정성 확보
- 실제 로봇 연동 대비 mission lifecycle, sensor registry, media profile 계약 검증 강화
- 6월에 확정한 event interface의 회귀 검증과 payload 확장 논의
- 이벤트/알람/탐지/AI/SOP thin-slice를 공통 event model 기반으로 정리
- 배포/운영/장애 대응 기본 체계 정리

## July Goals

1. 6월 PoC 기능을 반복 시연 가능한 관제 MVP로 안정화한다.
2. 6월에 확정한 `channel.event` interface가 구현, 저장, UI, 로봇팀 가이드에서 깨지지 않는지 검증한다.
3. 로봇팀 연동 계약을 stable 문서, 로봇팀 가이드, Swagger/OpenAPI 기준으로 확정한다.
4. 녹화/리플레이 실패, 재처리, 상태 표시를 운영 관점에서 정리한다.
5. 이벤트/알람, 탐지 결과, spatial 후보, AI/SOP PoC를 관제 화면 흐름에 자연스럽게 연결한다.
6. 배포/검증/장애 대응과 관제-로봇 로그 handoff 절차를 가이드 문서로 고정한다.

## Week 1: PoC Regression Hardening

기간: 2026-07-01 ~ 2026-07-07

목표: 6월 PoC 기능이 반복 실행해도 깨지지 않도록 회귀 테스트와 안정성을 보강한다.

작업:

- Python/GStreamer Mock Robot 기반 통합 테스트 정리
- 임무 생성/시작/종료/리플레이 smoke flow 보강
- Event interface 회귀 케이스 정리
  - `sensor.alarm`
  - `detection.object`
  - `robot.fault`
  - `communication.state`
  - malformed event payload
- 관제 화면 새로고침, robot reconnect, recorder-worker 재시작 검증
- live status, recording status, sensor latest 상태 불일치 케이스 정리
- mission lifecycle 회귀 케이스 정리
  - active mission 없음
  - mission start 후 publish
  - mission end 후 publish 중단
  - robot reconnect
  - 중복 publisher
- 실패/대기/처리 중 상태 UI 문구 일관화

완료 기준:

- dev stack 재시작 후 동일 시나리오를 반복 실행할 수 있다.
- 주요 smoke flow가 문서화된다.
- 회귀가 잦은 event/mission/WebRTC/recording 흐름에 테스트가 보강된다.

## Week 2: Real Robot Integration Readiness

기간: 2026-07-08 ~ 2026-07-14

목표: 실제 로봇팀이 연동하기 전에 관제 쪽 계약과 검증 도구를 고정한다.

작업:

- `docs/stable/robot-interface.md` 최종 점검
- Swagger/OpenAPI와 stable 계약 정합성 확인
- Event interface 호환성 점검
  - `channel.event` envelope
  - event type namespace
  - severity
  - source / position / media / payload
  - DB `events` mapping
  - UI event panel mapping
  - AI/SOP context mapping
- Robot Gateway token/mission/signaling flow 검증
- Mission lifecycle / always-on robot 계약 최종화
- Sensor type registry와 타입별 `values` shape 최종화
- Track/DataChannel sample payload와 media profile 정리
  - H.264/Opus
  - `msid` track id
  - DataChannel open 이후 send
  - 권장 fps/resolution/bitrate
- 로봇팀 테스트 가이드와 6월 event interface 정합성 점검
- GStreamer Mock과 Python Mock의 참조 구현 차이 정리
- 관제 로그와 로봇팀 리포트 매칭 checklist 정리

완료 기준:

- 로봇팀이 6월에 확정된 문서만 보고 heartbeat/mission 조회/WebRTC publish와 `channel.event` 송신을 구현할 수 있다.
- Mock Robot과 stable 문서가 동일한 계약을 따른다.
- 실제 로봇 연동 시 관제 쪽에서 확인할 checklist가 준비된다.
- 로봇팀이 새 센서, media profile, 장애 로그를 어떤 기준으로 맞춰야 하는지 문서만 보고 판단할 수 있다.

## Week 3: Recording Operations And Replay UX

기간: 2026-07-15 ~ 2026-07-21

목표: 녹화/리플레이를 운영 가능한 수준으로 안정화한다.

작업:

- 마지막 chunk finalize 실패 케이스 보강
- recording finalization job 재처리 정책 정리
- recorder-worker scale-out 시 중복 처리 방지 기준 정리
- DB recording metadata와 MinIO object 상태 검증
- H.264 녹화 실패 원인 표시와 codec/profile mismatch 진단 기준 정리
- replay 화면 UX 정리
  - robot별 session
  - chunk별 file
  - playable/unplayable 상태
  - 실패 원인 표시
- object storage URL 노출 방식 점검

완료 기준:

- 임무 종료 후 마지막 녹화가 재생 가능 상태로 마감된다.
- 실패한 녹화 파일은 UI와 DB에서 구분된다.
- media profile mismatch나 recorder 실패를 관제팀이 로그/화면으로 설명할 수 있다.
- `D-SRV-07`, `E-DB-06`을 done 후보로 검토할 수 있다.

## Week 4: Event Expansion, Spatial And AI/SOP PoC Integration

기간: 2026-07-22 ~ 2026-07-31

목표: 6월에 만든 event interface와 이벤트/AI thin-slice를 관제 흐름 안에서 설득력 있게 정리하고, 로봇팀이 추후 보낼 event/spatial payload의 경계를 정한다.

작업:

- 이벤트/알람 모델 확장
  - acknowledgement lifecycle
  - operator note
  - duplicate/suppression policy
  - event timeline filtering
- 탐지 결과 event 모델 정리
  - detection class
  - confidence
  - bounding box
  - world/map position 후보
  - related media timestamp
- `channel.spatial` 계약 후보 정리
  - frameId
  - coordinate frame
  - pose/odometry
  - point cloud object reference
  - 저장/표시 제외 범위
- Control / ACK는 구현하지 않고 command lifecycle과 ACK schema 초안만 정리
- AI/SOP 패널 PoC 정리
  - 상황 요약
  - 위험도 또는 우선순위
  - SOP 추천 placeholder/rule 기반 결과
  - event/sensor/detection context window
- PM/시연용 설명 문구와 기능 한계 정리
- feature 상태 재분류
  - `D-SRV-04`
  - `D-SRV-05`
  - `E-DB-04`
  - `E-DB-05`
  - `F-LLM-*`

완료 기준:

- 관제 화면에서 이벤트/알람/AI 요약이 임무 흐름과 연결되어 보인다.
- event interface가 `D-SRV-04`, `D-SRV-05`, `E-DB-04`, `E-DB-05`, `F-LLM-*`의 공통 입력 모델로 설명된다.
- detection event와 spatial payload는 로봇팀과 논의 가능한 초안 수준으로 정리된다.
- control/ack는 구현 범위가 아니라 후속 설계 범위로 분리되어 있다.
- 실제 LLM/탐지 모델 미연동 범위가 명확히 표시된다.
- 8월 작업으로 넘길 기능과 제품화 범위가 정리된다.

## Out Of Scope For July

7월에도 다음 항목은 제품 수준 구현에서 제외한다.

- 실제 로봇 제어 명령 운영
- `channel.control` command forwarding 운영
- 실제 AI 탐지 모델 성능 검증
- SOP 전체 rule set 구축
- 사용자 인증/권한/감사 로그 제품화
- TLS/암호화/보안 운영 전체 적용
- TURN ephemeral credential 운영 전환
- 3D SLAM/Point Cloud 제품 수준 시각화
- LiveKit 또는 외부 SFU 전환

## Expected Deliverables

- 관제 MVP 통합 검증 결과
- 로봇팀 연동 stable 가이드
- 6월 `channel.event` stable interface 회귀 검증 결과와 event storage/UI/AI mapping 보강
- mission lifecycle / sensor registry / media profile 확정안
- 관제-로봇 로그 handoff checklist
- 녹화/리플레이 운영 안정화 결과
- 이벤트/알람/탐지/AI/SOP PoC 화면
- spatial payload와 control/ack 후속 설계 초안
- recorder-worker 운영/scale-out 고려사항 문서
- 8월 제품화 우선순위 목록
