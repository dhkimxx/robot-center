# 2026-06 Monthly Plan

## Scope

기간: 2026-06-08 ~ 2026-06-30

6월 목표는 P0 기능 전체를 제품 수준으로 완성하는 것이 아니라, **P0 기능을 관제 PoC로 시연 가능한 thin-slice 형태로 완성**하는 것이다.

이번 달의 성공 기준은 다음 흐름을 끝까지 보여주는 것이다.

```text
임무 생성
-> 로봇 1~2대 연결
-> RGB/Thermal/GPS/센서 표시
-> 이벤트/알람 기본 표시
-> 녹화 중 상태 표시
-> 임무 종료
-> 녹화 리플레이 재생
-> 로봇팀 연동 문서 제공
```

## June Goals

1. 로봇팀 연동에 필요한 Robot Gateway/WebRTC 계약을 PoC 기준으로 확정한다.
2. `D-SRV-04/05`, `E-DB-04/05`, `F-LLM-*` 구현의 공통 기반이 되는 `channel.event` 인터페이스 v0를 이번 주에 정의하고 문서 확정한다.
3. 로봇 상시 대기, 임무 배정/시작/종료, 재연결, 중복 publisher 같은 mission lifecycle 계약을 고정한다.
4. 센서 데이터가 늘어나도 저장/표시가 깨지지 않도록 sensor type registry와 telemetry 확장 기준을 정리한다.
5. H.264/Opus media profile, track identity, DataChannel lifecycle, 관제-로봇 로그 handoff 기준을 검증 가능한 형태로 정리한다.
6. 이벤트/알람과 AI/SOP는 제품 수준이 아니라 PoC placeholder 또는 rule/mock 기반으로 구현한다.

## Week 1: Robot Contract, Event Interface And Live Pipeline

기간: 2026-06-08 ~ 2026-06-14

목표: 로봇팀이 붙을 수 있는 관제 연동 계약, features 공통 event interface, live pipeline을 이번 주 안에 고정한다.

작업:

- Robot Gateway API 정리
  - heartbeat
  - active mission 조회
  - SFU signaling URL
- Features 공통 Event Interface v0 정리
  - `channel.event` envelope
  - `eventType` namespace
  - severity
  - occurredAt / receivedAt 기준
  - source
  - media track 기준
  - 타입별 `payload`
- Event type v0 확정
  - `detection.object`
  - `mission.event`
- Event 저장/표시/AI 입력 매핑 정리
  - `detection.object`는 RGB/Thermal Live video overlay
  - `mission.event`는 Live event panel
  - DB 저장, 알림, replay sync, AI/SOP input context는 후속 범위
- 로봇 payload 경계 정리
  - robot payload에 `robotCode`, `missionCode`, `missionId`, `channelRole`을 넣지 않음
  - 서버가 token, room, DataChannel label 기준으로 context 주입
- Mission lifecycle 계약 정리
  - active mission 없음
  - mission 시작 후 publish 시작
  - mission 종료 후 publish 중단
  - robot reconnect
  - 같은 robot의 중복 publisher 처리
- WebRTC role별 endpoint 계약 정리
  - `/sfu/robot/ws`
  - `/sfu/operator/ws`
  - `/sfu/recorder/ws`
- Track/DataChannel 규칙 정리
  - `track.video_1`
  - `track.video_2`
  - `track.audio_1`
  - `channel.telemetry`
  - `channel.spatial`
  - `channel.event`
- WebRTC media profile 기준 정리
  - H.264/Opus 통과 기준
  - `msid` track id와 `mid`의 역할 분리
  - keyframe interval, resolution, fps, bitrate 권장값 후보
- 로봇팀 장애 리포트와 관제 로그 handoff 기준 정리
  - `robotCode`
  - `missionCode`
  - `peerId`
  - `messageId`
  - track id
  - DataChannel label
- Python/GStreamer Mock Robot 검증 흐름 정리
- `docs/stable/robot-interface.md`, `docs/stable/data-storage.md`, 로봇팀 가이드, Swagger/OpenAPI 정합성 확인

완료 기준:

- Mock Robot이 stable 문서 기준으로 관제에 붙을 수 있다.
- 관제 화면에서 선택한 로봇의 RGB/Thermal/GPS/센서가 표시된다.
- `channel.event` v0 계약으로 detection overlay와 mission event panel 표시를 설명할 수 있다.
- `channel.event` v0 계약이 stable 문서, 로봇팀 가이드, 저장 문서에 반영되어 추가 설계 없이 공유 가능하다.
- 로봇팀이 always-on robot 흐름에서 언제 publish를 시작/중단해야 하는지 문서만 보고 판단할 수 있다.
- `A-SYS-03`, `A-SYS-04`, `D-SRV-04`, `D-SRV-05`, `E-DB-04`, `E-DB-05`, `I-COM-02`, `I-COM-03`, `I-COM-04`, `F-LLM-02` 진행 근거가 문서에 반영된다.

## Week 2: Recording Replay And Sensor PoC

기간: 2026-06-15 ~ 2026-06-21

목표: 녹화/리플레이와 센서 저장/표시를 PoC 시연 가능한 수준으로 안정화한다.

작업:

- 임무 종료 시 마지막 recording chunk finalize 보강
- recorder-worker job 동시성/중복 처리 검증
- MP4 muxing 실패/재시도 정책 정리
- MinIO object 상태와 DB recording 상태 정합성 확인
- replay UI에서 robot/session/chunk/file 구조 정리
- SensorDescriptor/SensorSample 기반 latest sensor 표시와 초기 snapshot fallback 보강
- Sensor type registry PoC 확정
  - `position`
  - `battery`
  - `gas`
  - `imu`
  - `odometry`
  - `device_status`
- Gas module channel 해석 기준 정리
  - `CO`, `H2S`, `O2`, `CH4`, `TEMP`, `HUM`은 모두 `sensorType=gas`
  - `TEMP`, `HUM`도 같은 gas module의 channel label로 보고 `values.concentration`과 `descriptor.unit`으로 표시
  - 독립 온도/습도 센서가 필요하면 `temperature`, `humidity` 타입을 별도 후보로 논의
- 타입별 `values` shape와 표시 전략 정리
- sensor clear, retention, 장시간 테스트 데이터 증가 대응 기준 정리

완료 기준:

- 임무 시작 -> 로봇 송출 -> 녹화 -> 임무 종료 -> 리플레이 재생이 통과한다.
- 마지막 chunk가 `녹화 중`으로 남지 않는다.
- 센서 종류가 늘어나도 UI가 깨지지 않는다.
- 로봇팀이 새 센서 타입을 추가하기 전에 관제팀과 맞춰야 할 이름과 `values` 구조를 알 수 있다.
- `D-SRV-07`, `E-DB-03`, `E-DB-06`, `F-LLM-02` 진행 근거가 문서에 반영된다.

## Week 3: P0 PoC Feature Thin-Slice

기간: 2026-06-22 ~ 2026-06-28

목표: P0 후보 기능을 PoC에서 보일 수 있는 최소 단위로 구현한다.

작업:

- Event Interface v0 기반 이벤트/알람 기본 표시
  - DataChannel event 또는 mock event 수신
  - 관제 화면 이벤트 로그 표시
  - 경고/주의 상태 기본 chip 표시
- Event 저장 thin-slice
  - `events` table 저장
  - raw payload 보존
  - mission/robot context server-side 주입 확인
- 탐지 결과 시각화 thin-slice
  - 실제 AI 탐지 연동은 제외
  - 공통 event model 기반 mock detection event를 영상/이벤트 패널에 표시
- AI/SOP thin-slice
  - 실제 LLM 품질 검증은 제외
  - sensor/event/detection context 기반 rule/mock 상황 요약 또는 추천 패널 표시
- 관제 화면 UI 정리
  - 로봇 선택
  - 송출/녹화/연결 상태
  - 영상 overlay
  - 센서/이벤트 영역

완료 기준:

- 관제 화면에서 영상/센서/이벤트/기본 AI 요약 영역이 한 흐름으로 보인다.
- mock event/detection으로 P0 기능 방향성을 시연할 수 있다.
- event interface를 바꾸지 않고 detection overlay와 mission event panel mock을 모두 표현할 수 있다.
- `D-SRV-04`, `D-SRV-05`, `E-DB-04`, `E-DB-05`, `F-LLM-*` 중 실제 착수된 항목은 `planned -> in-progress` 후보로 검토한다.

- 실제 긴급 제어 명령
